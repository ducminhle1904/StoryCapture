<objective>
Ship Windows.Graphics.Capture parity for the core "record a specific window" path on Windows, using `windows-capture = "=2.0.0"`. Plan 05-01 already defines the `CaptureBackend` trait surface, `CaptureTarget` enum, and the silent-xcap-fallback orchestrator; this plan only fills in the Windows-side backend + window enumeration + pid→HWND resolution behind them.

Purpose: Windows users get the same "record only the demo browser" capability as macOS users. The trait abstraction was built in Phase 1 precisely so this plan can be a surgical fill-in, not a cross-cutting redesign.
Output: Working WGC backend, Windows window enumeration IPC, Playwright-auto on Windows, CI build gate so Windows can't regress silently.
</objective>

From `windows-capture = "=2.0.0"`:
- `windows_capture::window::Window::enumerate() -> Result<Vec<Window>, _>`
- `windows_capture::window::Window::from_raw_hwnd(isize) -> Window`
- `windows_capture::window::Window::from_name(&str) -> Result<Window, _>`
- `windows_capture::window::Window::foreground() -> Result<Window, _>`
- `windows_capture::monitor::Monitor::enumerate() -> Result<Vec<Monitor>, _>` (display path)
- `windows_capture::capture::GraphicsCaptureApiHandler` trait — 2.0's handler trait (renamed from 1.x)
- `windows_capture::settings::Settings::new(capture_item, ...)` — generic over item type via `TryInto<GraphicsCaptureItemType>`
- `windows_capture::frame::Frame` — exposes D3D11 texture; `.buffer()` for CPU read-back

From Plan 05-01:
- `CaptureBackend` trait (unchanged)
- `CaptureTarget { Display, Window { WindowId }, WindowByPid { pid, title_hint } }`
- The silent-fallback orchestrator in `crates/capture/src/lib.rs` — we plug `WgcBackend` behind it via `cfg(target_os = "windows")` the same way `SckBackend` plugs in on macOS.

PID → HWND on Windows (RESEARCH.md §Don't Hand-Roll):
```rust
// Preferred: filter Window::enumerate() by pid
use windows_capture::window::Window;
// If Window exposes .process_id() (verify in 2.0 docs), one-liner:
let hwnd_opt = Window::enumerate()?.into_iter()
    .find(|w| w.process_id() == Some(pid));

// Fallback: use `windows` crate EnumWindows + GetWindowThreadProcessId.
// (Already in Cargo.lock at 0.58 per STACK.md.)
```

Open Question 4 (RESEARCH.md): Chromium on Windows uses a browser parent process + sandboxed renderer children. Top-level windows live on the browser PID (which is what `browser.process().pid` returns from Playwright). If enumeration misses, walk child processes via `windows::Win32::System::Diagnostics::ToolHelp::CreateToolhelp32Snapshot` + `Process32First/Next`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frontend → Tauri IPC → Rust capture (Windows) | `CaptureTarget::Window { WindowId }` carries an isize HWND across privilege boundary |
| Rust capture → `windows::Win32::*` FFI | HWND validity + pointer safety; bad HWND could cause UB in the `windows` crate calls — but `windows` crate wraps these safely |
| Rust capture → `windows-capture` 2.0.0 | 3-days-old crate; internal soundness not yet community-audited |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-03-01 | Tampering / EoP | `CaptureTarget::Window { window_id }` — malicious renderer passes an HWND of another app (e.g. a banking window) to capture it | mitigate | HWND must appear in the most recent `list_windows()` result to be accepted — applied in Task 3. list_windows already filters to visible + owned-by-visible-pid windows. |
| T-05-03-02 | InfoDisclosure | Window titles in logs | mitigate | TRACE-only logging (parity with Plan 05-01 T-05-01-02). |
| T-05-03-03 | EoP | Cross-session window capture | mitigate | Windows' `EnumWindows` returns only the current desktop's windows by default; additionally filter to windows whose owning process is in the current session (PID-to-session via `ProcessIdToSessionId`). Apply in Task 2 window enumeration. |
| T-05-03-04 | T/I | WGC permission model (Windows 10 19H1+ — no user prompt for WGC, unlike macOS TCC) | accept | WGC grants are implicit on supported Windows versions; no mitigation needed. Document in SUMMARY. |
| T-05-03-05 | D | `windows-capture` 2.0.0 soundness bug causes panic | accept | Caught by `std::panic::set_hook` (Phase 1) → user-visible restart dialog, no silent failure. Pin exact version `=2.0.0` to avoid drift. |
| T-05-03-06 | I | Frame buffer read-back leaks across processes (D3D11 staging texture lifecycle) | mitigate | Use wrapper-provided buffer extraction only; do not hand-roll D3D11 Map/Unmap. Applied in Task 1 via `frame.buffer()`. |
| T-05-03-07 | Tampering | Chromium child-process-walk (Open Question 4 fallback) could match a non-Chromium child of the browser PID | mitigate | After matching by pid tree, also verify the target window's class name starts with `"Chrome"` or its process name is `"chrome.exe"` / `"msedge.exe"`. Applied in Task 2. |
</threat_model>
