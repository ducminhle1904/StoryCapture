<objective>
Bridge Playwright's Node sidecar to macOS ScreenCaptureKit so StoryCapture automatically records the Chromium window that Playwright just launched, with no user picking required. Plan 05-01 already ships the generic `CaptureTarget::WindowByPid` variant and picker UI; this plan fills in the pid source (new sidecar verb) and the pid→SCWindow resolver with retry to handle Chromium's launch→window-register race.

Purpose: Deliver the "Playwright browser (auto)" default that makes the product feel effortless — D-01 says this is the first-run default, D-02 says it's the recommended option. This plan makes the default actually work.
Output: Playwright launch → pid resolution → capture auto-selects the matched window; end-to-end smoke binary proves the path on CI.
</objective>

Playwright sidecar JSON-RPC verb (new):
```json
// Request: { "id": 42, "method": "browserProcess", "params": {} }
// Response (local): { "id": 42, "result": { "pid": 12345, "executablePath": "/path/to/Chromium" } }
// Response (remote browser): { "id": 42, "result": { "pid": null, "executablePath": null, "reason": "remote-browser" } }
// Response (no browser launched): { "id": 42, "error": { "code": -32000, "message": "browser not launched" } }
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Playwright sidecar (Node child process) → Rust host (JSON-RPC over stdio) | Sidecar is spawned by us and runs with our privileges; malicious input could only originate from the sidecar binary we ship. Low risk, still validate. |
| Rust host → user UI | `executablePath` and `pid` returned to the UI; pid is an integer, executablePath may be a user-home path |
| UI → Rust `start_capture` (Playwright-auto path) | `CaptureTarget::WindowByPid { pid, title_hint }` crosses from renderer to host; pid and title_hint must be validated |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02-01 | Tampering | `CaptureTarget::WindowByPid { pid }` over IPC — malicious renderer supplies a pid of a non-Playwright app to capture it | mitigate | On the Playwright-auto code path, the pid MUST come from `browser_process()` inside the host, never from a round-trip through the renderer. Renderer asks "use Playwright auto"; host builds the CaptureTarget. Applied in Task 3. |
| T-05-02-02 | Tampering | `title_hint` injection — overly long / control-char strings | mitigate | Reject non-ASCII control chars; cap 256 chars. Applied in Task 3. |
| T-05-02-03 | InfoDisclosure | `executablePath` (may contain `/Users/<name>/…`) in logs/telemetry | mitigate | Log at DEBUG only; never in `tauri-plugin-log` INFO output; the project has no telemetry so external leak is already blocked. Applied in Task 1. |
| T-05-02-04 | InfoDisclosure | Window titles logged when resolving pid→window (e.g. "Gmail — Personal") | mitigate | Inherits Plan 05-01 T-05-01-02 mitigation (TRACE-only). Applied via reuse of `find_window_by_pid`. |
| T-05-02-05 | EoP | TCC prompt during auto-flow tricks user into granting to wrong app | accept | TCC dialog is OS-controlled and shows our bundle identity; can't be spoofed by us. |
| T-05-02-06 | D (DoS) | Sidecar returns rapid-fire `browserProcess` calls during UI refresh | mitigate | Debounce `refreshPlaywrightAvailability` in Zustand to ≤1 call/s. Applied in Task 3. |
| T-05-02-07 | S | Fake sidecar binary replaced on disk | accept | Sidecar is shipped inside the signed/notarized app bundle; replacement implies the bundle is already compromised. OS code-signing enforcement covers. |
</threat_model>
