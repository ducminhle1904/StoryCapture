# Phase 09-04: Editor-surface Live Preview + viewport switcher

## Purpose

Extend the Phase 9 live-preview pipeline so it renders in **two places**: the recording view (already shipped in 09-02/03) and the Editor page's preview panel. Add an author-time viewport switcher that drives Playwright's viewport on the fly. This makes the Editor page's preview-panel.tsx a functional authoring tool instead of a static placeholder.

## Design summary

- **Reuse, don't fork.** `LivePreview.tsx` becomes a pure presentational component keyed by a `streamId`. The recorder passes its recording session id; the editor passes the author-time session id. The canvas renderer is identical.
- **Multi-stream sidecar.** Phase 9 today assumes one preview stream at a time. We extend `startPreviewStream` to accept a `streamId` parameter; the sidecar keeps a `Map<streamId, SessionState>` with per-stream latest-frame buffer. Rust `subscribe_preview(streamId)` returns a per-stream `watch::Receiver`.
- **Ephemeral author session.** New Tauri commands `start_author_preview` / `stop_author_preview`:
  1. Spawn a fresh Playwright sidecar instance (or reuse via ref-count)
  2. `launch` with `meta.app` as the initial URL and the current Editor-panel viewport
  3. Call `startPreviewStream` with a fresh UUID streamId
  4. Store the streamId on the editor state
- **Viewport switcher.** The existing three-option switcher in `preview-panel.tsx` already has labels. We wire the handler to `invoke('set_author_preview_viewport', { width, height })`, which forwards to `setViewport` RPC → `page.setViewportSize()`. No page reload required.
- **Cold-start budget.** Default `previewEnabled=false`. A prominent but dismissable "Preview on / off" toggle in the Editor panel header. Tooltip: "Launches a hidden Chromium to mirror your story as you edit."
- **Teardown.** React effect cleanup calls `stop_author_preview` on unmount / page navigation / toggle-off. Sidecar kills the associated Chromium process and frees the streamId entry.

## Interaction with in-flight recording

- Recording view and editor preview can coexist. They consume different streamIds, different Chromium processes.
- Switching from the Editor page to the recorder does NOT tear down the author session; it parks in the background at 0 fps until the panel is visible again (listen to `IntersectionObserver` + tab visibility).

## Out of scope

- Running DSL steps against the author-time session (Phase 10).
- Synchronizing DSL edits into the author-time session's URL (user must click "Reload" or the session navigates on explicit `meta.app` change).
- Input forwarding into the preview canvas (same as Phase 9's watch-only constraint).

## Risk flags

- **Two Chromium processes is double the memory.** M2 baseline ≈ 200 MB per headless Chromium. Author session + recording session peak ≈ 400 MB. Within PROJECT.md's <800 MB recording budget but tight. Mitigation: default-off toggle; consider "one Chromium, two CDP targets" optimization as a follow-up if telemetry shows concern.
- **Existing Phase 9 tests assume single stream.** Plan 09-01/02/03 tests need to be updated to pass `streamId` explicitly and cover the multi-stream matrix. Fold into this plan's test budget, not a separate one.
- **Sidecar process lifecycle.** If the author session's sidecar crashes, the recording sidecar must be unaffected. Run them as **separate** Node SEA child processes keyed by purpose; do NOT share a single sidecar between author and recording roles.
