    - Foreign keys ON (`PRAGMA foreign_keys = ON` on every connection)
    - WAL mode enabled (`PRAGMA journal_mode = WAL`)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User filesystem → ProjectDb | Opened project folders may be tampered (partial SQLite, truncated files). |
| `app.sqlite` → IPC → UI | Project list returned via typed IPC; no raw paths or secrets leak. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | Corrupted SQLite file | mitigate | Open failure returns `StorageError::Sqlite(...)`; UI surfaces error; WAL mode provides some durability. |
| T-05-02 | Tampering | User manually edits schema version | mitigate | `SchemaVersionMismatch` prevents writes on mismatch; no silent data corruption. |
| T-05-03 | Information Disclosure | API keys stored in SQLite | mitigate | Explicit policy: secrets go to keyring (Plan 03), not SQLite. No column named `*_key`, `*_secret`, `*_token` in schema. |
| T-05-04 | Denial of Service | Gigabyte project.sqlite | accept | SQLite handles multi-GB; PRAGMA size enforcement deferred to Phase 5. |
| T-05-05 | Elevation of Privilege | Project folder containing `project.sqlite` with malicious schema | mitigate | Migrations verify user_version; foreign key violations caught by `PRAGMA foreign_keys=ON`. |
</threat_model>
