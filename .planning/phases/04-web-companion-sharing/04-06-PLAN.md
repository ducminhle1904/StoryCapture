    1. Create trpc/routers/workspace.ts with RBAC middleware per RESEARCH.md:

       Define middleware procedures (not exported, internal to router):
       - `workspaceMemberProcedure`: extends protectedProcedure, validates user is a member of the workspace (input must include workspaceId). Adds `membership` to context.
       - `workspaceEditorProcedure`: extends workspaceMemberProcedure, throws FORBIDDEN if role is VIEWER
       - `workspaceOwnerProcedure`: extends workspaceMemberProcedure, throws FORBIDDEN if role is not OWNER

       Procedures:
       `create` (protectedProcedure, mutation):
       - Input: `{ name, slug? }` (auto-generate slug from name if not provided)
       - Create workspace + add creator as OWNER
       - Validate slug uniqueness

       `list` (protectedProcedure, query):
       - Return all workspaces where user is a member, with their role and member count

       `getById` (workspaceMemberProcedure, query):
       - Return workspace details including members list and video count

       `update` (workspaceOwnerProcedure, mutation):
       - Input: `{ workspaceId, name?, slug? }`
       - Only owner can rename or change slug

       `delete` (workspaceOwnerProcedure, mutation):
       - Only owner can delete; cannot delete personal workspace (isPersonal: true)
       - Cascade deletes members, invites (videos deletion is a separate concern — warn user)

       `invite` (workspaceEditorProcedure, mutation):
       - Input: `{ workspaceId, email, role }` (per D-04: invite by email)
       - Owner can invite with any role; editor can invite as editor or viewer only
       - Create WorkspaceInvite with random token + 7-day expiry
       - Send invite email via Resend (`resend` package added in Plan 04-01):
         - Import from `@/lib/email` (created in Task 2 below)
         - Call `sendInviteEmail(email, inviteLink, workspaceName, inviterName)`
         - If RESEND_API_KEY is not configured, gracefully degrade to returning the invite link only (user copies and shares manually)
         - Email template: simple HTML with workspace name, inviter name, "Accept Invite" CTA button linking to /invite/<token>

       `acceptInvite` (protectedProcedure, mutation):
       - Input: `{ token }`
       - Validate token exists, not expired
       - Add user as member with specified role
       - Delete the invite token (consumed)

       `removeMember` (workspaceOwnerProcedure, mutation):
       - Input: `{ workspaceId, userId }`
       - Cannot remove self if sole owner
       - Delete membership

       `updateMemberRole` (workspaceOwnerProcedure, mutation):
       - Input: `{ workspaceId, userId, role }`
       - Only owner can change roles

       `leave` (workspaceMemberProcedure, mutation):
       - Input: `{ workspaceId }`
       - Cannot leave if sole owner (must transfer ownership first)

    2. Merge workspaceRouter into _app.ts

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated user -> workspace procedures | Role-based access control gates all operations |
| Public -> /invite/[token] | Token-bearer can join workspace with assigned role |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-21 | E (Elevation) | RBAC bypass | mitigate | 3-tier middleware (member/editor/owner) checked on every procedure; editor cannot assign owner role |
| T-04-22 | S (Spoofing) | invite token | mitigate | Random CUID token, 7-day expiry, consumed on use (single-use) |
| T-04-23 | D (DoS) | invite spam | accept | No rate limiting on invites in v1; invites are link-based (no email sending) so low abuse vector |
</threat_model>
