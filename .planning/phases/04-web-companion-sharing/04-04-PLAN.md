       - `createPresignedPartUrl(bucket, key, uploadId, partNumber)`: returns presigned PUT URL (1 hour expiry)
       - `createPresignedGetUrl(bucket, key)`: returns presigned GET URL (1 hour expiry) for private reads
       - Export `R2_BUCKET` from env

       - CRITICAL per Pitfall 3: presigned URLs use S3 API domain only, NOT custom domain

    2. Create trpc/routers/video.ts with these procedures (per RESEARCH.md code example):

       `initiateUpload` (protectedProcedure, mutation):
       - Input: `{ fileName, fileSizeBytes, contentType, workspaceId, projectName, storySource?, sceneBoundaries? }`
       - Verify user has EDITOR or OWNER role in workspace
       - R2 key: `{workspaceId}/{uuid}/{fileName}` (hierarchical per RESEARCH.md R2 Bucket Structure)
       - Call `CreateMultipartUploadCommand` with `ServerSideEncryption: 'AES256'` (DIST-06)
       - Create Video record with status UPLOADING, generate slug from projectName (slugify + uniqueness check)
       - Return `{ videoId, uploadId, r2Key, slug }`

       `getPartPresignedUrl` (protectedProcedure, mutation):
       - Input: `{ r2Key, uploadId, partNumber }`
       - Verify user owns the in-progress upload
       - Return presigned PUT URL
       - CRITICAL per Pitfall 2: validate partNumber >= 1, <= 10000

       `completeUpload` (protectedProcedure, mutation):
       - Input: `{ videoId, r2Key, uploadId, parts: [{ PartNumber, ETag }], thumbnailR2Key: string }` (thumbnailR2Key is an explicit input parameter from the desktop, not derived server-side -- warning fix)
       - Call `CompleteMultipartUploadCommand`
       - Update Video record: set status to READY and thumbnailR2Key from the explicit input
       - Return `{ videoId, slug, status: 'READY' }`

       `abortUpload` (protectedProcedure, mutation):
       - Input: `{ videoId, r2Key, uploadId }`
       - Call `AbortMultipartUploadCommand`
       - Update Video status to FAILED

       `list` (protectedProcedure, query):
       - Input: `{ workspaceId }`
       - Return videos in workspace with status, slug, thumbnail URL, creation date
       - Generate presigned GET URLs for thumbnails

       `getBySlug` (publicProcedure, query):
       - Input: `{ slug }`
       - Return video if READY (public data only: projectName, sceneBoundaries, video URL)
       - For public videos: return R2 public URL. For private (unlisted): return presigned GET URL

    3. Merge videoRouter into _app.ts

    4. Create the 3 REST upload endpoints (app/api/upload/) as thin wrappers around the tRPC procedures. These exist for the desktop Rust client which may prefer direct HTTP calls over tRPC protocol:
       - initiate/route.ts: POST, validates desktop JWT from Authorization header, calls initiateUpload logic
       - presign/route.ts: POST, validates JWT, returns presigned part URL
       - complete/route.ts: POST, validates JWT, completes upload
       All use `verifyDesktopToken` from lib/jwt.ts for auth (desktop uses JWT, not session cookie)

       - Mints a short-lived JWT by calling the web companion (or reuse the stored token if it acts as bearer)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Desktop -> Web API | Desktop JWT auth for upload orchestration |
| Desktop -> R2 | Presigned PUT URL (time-limited, part-specific) |
| Web API -> R2 | Server-side S3 operations with R2 credentials |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-12 | S (Spoofing) | Upload auth | mitigate | Desktop JWT verified on every upload API call; JWT is short-lived (15 min) |
| T-04-13 | T (Tampering) | Presigned URL | mitigate | URLs are part-specific (partNumber encoded), time-limited (1 hour), and signed by server |
| T-04-14 | I (Info Disclosure) | R2 credentials | mitigate | R2 access keys only on server (Vercel env vars); never sent to client. Presigned URLs are the only client-facing artifact. |
| T-04-15 | D (DoS) | Large uploads | accept | No file size limit in v1 — R2 supports up to 5 GiB per object. Workspace-level quotas deferred to v2. |
| T-04-16 | I (Info Disclosure) | Data at rest | mitigate | SSE-S3 AES256 encryption on all R2 objects per DIST-06 |
</threat_model>
