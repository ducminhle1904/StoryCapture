    1. Create commands/web_sync.rs with Tauri commands:

       `sync_project_metadata`:
       - Input: project ID (desktop UUID), project name, story source text
       - Reads web API token from keychain
       - Mints short-lived JWT (or uses token directly)
       - Calls web companion `sync.pushMetadata` via HTTP POST to `/api/trpc/sync.pushMetadata`
       - On success: update local sync timestamp in project.sqlite
       - On failure (no internet): queue the update in a `sync_queue` table in app.sqlite
         - Schema: `id, desktop_id, payload_json, created_at, status ('pending'|'sent')`
       - Per D-07: offline queue in SQLite, flush on reconnect

       `update_recording_status`:
       - Input: project ID, status string ("idle" | "recording" | "step:N/M")
       - Same HTTP call pattern as sync_project_metadata but to `sync.updateRecordingStatus`
       - Fires frequently during recording — use fire-and-forget (don't block recording)
       - On failure: don't queue (recording status is ephemeral, stale data not useful)

       `flush_sync_queue`:
       - Reads all pending items from sync_queue table
       - Sends each via HTTP to the web companion
       - On success: mark as 'sent' and delete
       - On failure: leave as 'pending' for next attempt
       - Called on app startup and on network reconnect

       `get_sync_status`:
       - Returns: connected (bool), pending queue count, last sync timestamp

    2. Wire recording status updates into the existing recording pipeline:

       - On recording start: send "recording"
       - On each step: send "step:N/M"
       - On recording complete: send "idle"
       - Only sends if a web account is connected (check keychain)
