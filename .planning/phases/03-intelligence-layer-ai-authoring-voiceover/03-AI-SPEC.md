## 1. System Classification

**System Type:** Hybrid — Structured Extraction (natural-language → pest DSL) + Conversational (per-step regen loop) + Content Generation (TTS narration script + audio synthesis).

**Description:**
A local desktop AI layer inside StoryCapture (Tauri v2, Rust core) that lets non-developers describe a software demo in plain English and receive a grammar-conformant `.story` DSL file with a per-step diff preview they can approve, edit, regenerate, or reject. The same intelligence layer generates AI voiceover (TTS) from an auto-script derived from DSL steps and snaps clips to DSL step boundaries. A secondary surface exposes an in-process LSP (tower-lsp) that feeds CodeMirror hover/autocomplete/diagnostics from the same pest grammar. "Good" = the proposed DSL parses on first try, every step maps 1:1 to a user-intent sentence, selector heuristics surface real risks not false alarms, and TTS clips land within ±50 ms of step boundaries. All keys live in the OS keychain; no prompts/responses ever leave the chosen provider.

**Critical Failure Modes:**
1. **DSL grammar non-conformance** — LLM returns text that fails pest parse, leaving the user stuck at an error dialog instead of an editable diff.
2. **Selector hallucination** — LLM invents CSS selectors for elements the page does not have, breaking `.story` execution silently at run-time on the user's machine.
3. **Step boundary drift** — TTS clip duration ≠ step duration mapping, causing voiceover to overrun into the next scene or cut off mid-word.
4. **Prompt-cache miss cascade** — System prompt/grammar not wrapped with `cache_control` so token cost balloons per turn; kills the cost guardrail's credibility.
5. **Secret leak** — API key written to `project.sqlite`, log file, or error toast text; violates AI-05 / D-20 and ships user credentials to crash reports.

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Developer tooling / DevRel content production — specifically authoring of polished software demo videos for SaaS marketing, sales enablement, onboarding, and technical tutorials. Adjacent to "demo automation" tools (Arcade, Supademo, Tella, Screen Studio) but content-producing rather than browser-replay.

**User Population:**
- **Solo founders & technical marketers** không có kỹ năng video editing — cần demo nhanh để thay thế Loom/Screen Studio làm tay.
- **DevRel engineers & PMMs** — cần lặp lại demo cho mỗi release một cách nhất quán.
- **Sales engineers** — cần snippet demo customizable cho từng prospect.
- **Content creators / educators** — sản xuất tutorial chất lượng cao.

Common thread: practitioner viết kịch bản, không phải video editor. Họ đánh giá output bằng tiêu chí *"liệu tôi có dám gửi cái này cho khách hàng / publish lên YouTube không?"*

**Stakes Level:** **Medium.** Single-user desktop tool, không liên quan tính mạng hay tài chính. Tuy nhiên output là **customer-facing marketing asset**: lỗi lãng phí record cycle (vài phút mỗi lần), làm hỏng brand voice, hoặc tệ hơn là **leak confidential data** chụp được trên màn hình test (pre-launch features, customer names, internal URLs, API keys hiển thị trong UI).

**Output Consequence:**
DSL do AI sinh ra được thực thi **thật** trên `chromiumoxide` ở máy user, record screen, encode MP4, rồi user post lên landing page / Twitter / sales deck. Một selector sai = wasted record cycle. Một verb hallucinated = grammar fail, user nhận error dialog thay vì diff editable. Một câu narration TTS phát âm sai brand name = perceived unprofessional, có thể phải re-render toàn bộ. Sai timing TTS = voiceover overrun sang scene kế, người xem nghe seam rõ rệt.

### What Domain Experts Evaluate Against

**Dimension: DSL grammar conformance (first-try parse rate)**
- Good: Output `.story` parse sạch qua pest grammar lần đầu, không cần user fix manually. Mỗi verb tồn tại trong verb catalog, mỗi step có đúng số argument. Practitioner đọc xong nói "này chính là DSL tôi sẽ tự gõ nếu tôi rành cú pháp."
- Bad: Output có verb không tồn tại (`hover_over`, `fill_form` khi catalog chỉ có `hover`, `type`), thiếu argument bắt buộc, hoặc syntax error khiến parser fail. User thấy red squiggle thay vì diff editable.
- Stakes: **Critical** — đây là gateway của toàn bộ trải nghiệm NL Mode. Match Critical Failure Mode #1 trong Section 1.
- Source: Pest grammar nằm tại `crates/story-parser` (Phase 1). Code-LLM hallucination research (Liu et al. 2025) chỉ ra "Functional Requirement Violation" và fabricated API/library calls là failure mode chiếm tỷ lệ cao nhất khi LLM sinh DSL/code.

**Dimension: 1-to-1 step mapping fidelity**
- Good: Mỗi câu user mô tả hành động → đúng 1 DSL step, đúng thứ tự, không gộp, không drop. Per-step diff card trong UI-07 maps thẳng vào câu tương ứng trong prompt; user có thể trỏ vào step nào và nói "câu này tôi mô tả ở đoạn nào".
- Bad: LLM gộp "click Sign Up rồi điền email" thành 1 step `click_and_type`, hoặc drop step "scroll xuống pricing section" khi prompt dài. Per-step regen trong card sai vị trí step.
- Stakes: **High** — vi phạm contract của UI-07 (per-step approve/edit/regenerate/reject).
- Source: DevRel practitioners mô tả demo theo "beat" — mỗi beat = 1 user-visible action. Gộp = mất khả năng iterate per-step.

**Dimension: Selector groundedness (no fabricated selectors)**
- Good: Selector LLM đề xuất hoặc (a) generic + có fallback strategy hợp lý (text-based, role-based, nth-of-type), hoặc (b) LLM **proactively flag ambiguity**: "Bạn nói 'blue Sign Up button' nhưng tôi không có DOM context — đề xuất `button:has-text('Sign Up')` và bật `wait_for_visible`, dry-run để xác nhận."
- Bad: LLM sinh selector cụ thể như `.btn-primary-xl#signup-2024` mà page không có element nào match — hoặc tệ hơn, match nhầm element. Failure phát hiện chỉ khi record run thật.
- Stakes: **Critical** — match Critical Failure Mode #2 trong Section 1. Đây là failure mode kinh điển của UI-automation LLM tools (Library Hallucinations 2025: pre-2025 prompts hallucinate non-existent libraries up to 84%).
- Source: arXiv 2509.22202 (Library Hallucinations); chromiumoxide selector behavior; D-17 selector heuristic warnings trong LSP.

**Dimension: Narration ↔ step semantic faithfulness**
- Good: Mỗi câu narration của step N **mô tả đúng hành động và outcome của step N**. Không invent UI state ("our beautiful pricing page" khi step thực ra navigate đến `/features`). Không reference step khác đã thay đổi sau regen ("the modal we opened earlier" khi modal đã bị remove).
- Bad: Script TTS chứa claim không có trong DSL (giá tiền, feature name không xuất hiện, social proof bịa). Per-step regen làm step N+1 reference step N cũ.
- Stakes: **High** — narration sai ship vào marketing asset = misleading advertising risk + brand damage.
- Source: Content faithfulness rubric (ai-evals.md); DevRel PMM practice — script phải verifiable từ demo flow.

**Dimension: TTS pronunciation & brand fidelity**
- Good: Brand names, CLI commands, product terms, technical acronyms phát âm đúng (qua phonetic hints / pronunciation dictionary). Voice giữ nhất quán xuyên suốt — không có "robot seam" giữa các per-step clip concatenated. Tone match brand preset user chọn.
- Bad: TTS đọc "Kubric" thành "Cuh-brick", "kubectl" thành "cube-cuttle", "API" thành "ah-pee". Volume/pitch jump giữa clips. Vocal fry/uptalk khi brand chọn "professional confident".
- Stakes: **High** — single mispronunciation của product name khiến demo bị publish lại.
- Source: Cartesia/Coval TTS evaluation criteria (pronunciation accuracy, naturalness, stability, brand alignment); WellSaid Labs naturalness research.

**Dimension: TTS ↔ step boundary timing precision**
- Good: TTS clip ends within ±50ms của step boundary. Per D-13, step duration **co giãn theo TTS** (TTS = ground truth): clip dài hơn step → step kéo dài bằng pause/freeze frame; ngắn hơn → silence padding cuối. Không overflow, không cut mid-word.
- Bad: TTS overrun sang scene/transition kế gây nhiễu. Cut giữa câu vì step duration cố định. BGM auto-duck (Phase 2 D-22) trigger sai thời điểm.
- Stakes: **Critical** — match Critical Failure Mode #3.
- Source: Cinematic post-production practice; D-13 sync rule; BGM duck dependency Phase 2 D-22.

### Known Failure Modes in This Domain

1. **Phantom selector hallucination** — LLM tạo CSS selector trông plausible (`.btn.primary[data-testid="signup"]`) nhưng page không có. User chỉ phát hiện khi record thật, mất 3-10 phút mỗi lần. *Mitigation hook: D-17 LSP selector heuristic warning + D-19 Dry-Run feedback loop.*

2. **Step merger / silent step drop on long prompts** — Prompt > 200 từ, LLM gộp 2 hành động thành 1 step hoặc drop step ở giữa. Particularly bad cho PMM-written briefs (verbose). *Mitigation hook: prompt template force "one user-visible action per step", post-validation count steps vs. action verbs in prompt.*

3. **Cross-step regen context drift** — User regen step 3 → step 5 narration vẫn reference "the modal from step 3" theo flow cũ. Voiceover script becomes nonsensical sau vài turn iterate. *Mitigation hook: per-step regen prompt phải re-feed full updated `.story` + flag dependent narrations để recompute.*

4. **TTS brand-name mispronunciation** — "Kubric" (silent K), "kubectl" (kube-control), domain names, acronyms đọc theo phonetic literal. *Mitigation hook: per-project pronunciation dictionary cho TTS providers hỗ trợ (ElevenLabs SSML phoneme tags, OpenAI prompt prefix workaround).*

5. **TTS clip seam between per-step clips** — Concatenate clips từ same voice nhưng different generation runs → audible volume/pitch jump. Listener perceive "robot stitched". *Mitigation hook: cùng voice_id + voice_settings (stability, similarity_boost) cho all clips của project; loudness normalize pass post-concat.*

6. **Prompt-cache invisible miss** — User edit grammar file hoặc verb catalog → invalidate Anthropic `cache_control` block; cost 5× tăng âm thầm. *Mitigation hook: D-04 token counter + warning khi cache hit ratio drop dưới ngưỡng (vd 50%).*

7. **Confidential-data leak qua LLM prompt** — User prompt chứa real customer URL, internal-only feature flag, API key visible trong UI screenshot. *Mitigation hook: PROJECT.md no-telemetry default; explicit copy trong onboarding "your prompts go to the chosen LLM provider — do not paste secrets". Future: opt-in regex redaction pre-send.*

### Regulatory / Compliance Context

Single-user desktop tool, **không** chạm HIPAA / SOX / FINRA / COPPA / PCI. Các ràng buộc trực tiếp áp dụng:

- **EU AI Act — Article 50 transparency obligations (effective 2 August 2026).** Output AI-generated voiceover + AI-generated DSL→video là "limited-risk AI" với **disclosure requirement**: nếu user distribute video ra ngoài (đặc biệt EU audience), họ — chứ không phải StoryCapture — có obligation đảm bảo audience biết content có yếu tố AI-generated. Audio modality requires audible signal hoặc machine-readable marker. **Action cho Phase 3:** (a) export pipeline có optional setting "embed AI-generated marker (C2PA-style metadata)", (b) onboarding copy cảnh báo user về EU AI Act obligation, (c) nhắc rằng disclosure trách nhiệm cuối cùng thuộc về user khi distribute. Code of Practice guidelines publish Q2 2026.

- **GDPR (Article 28 — data processor relationship).** Khi user paste prompt chứa personal data, LLM API (Anthropic/OpenAI) trở thành data processor cho user. StoryCapture là data controller helper. **Action:** privacy copy onboarding nói rõ "prompts/scripts được gửi tới {provider you chose}; review their DPA"; D-20 keychain đảm bảo credentials không leak; no-telemetry default đảm bảo StoryCapture tự nó không thấy prompt content.

- **Trade secret / IP confidentiality.** Prompts thường chứa pre-launch features, customer names, internal product roadmap. Không phải regulation nhưng là enterprise procurement blocker. **Action:** no-telemetry default (đã có trong PROJECT.md); cân nhắc opt-in "self-hosted LLM via Ollama" cho enterprise — defer post-Phase-3.

- **WCAG 2.1 AA (PROJECT.md baseline).** Chat panel, per-step diff cards, voice catalog browse, LSP hover popovers — tất cả surface mới phải accessible: keyboard nav, focus management, screen reader cho diff cards, ARIA live region cho streaming LLM response.

Không áp dụng: HIPAA (no PHI), SOX/FINRA (no financial advice), COPPA (no minors targeting), CCPA-as-service (single-user local, không "sale of data").

## 2. Framework Decision

**Selected Framework:** Direct HTTP stack in Rust — `reqwest 0.12` (rustls) + `eventsource-stream 0.2` + `serde/serde_json 1.x` + `tokio 1.40+`, wrapped by hand-written `LlmProvider` and `TtsProvider` traits in a new `crates/intelligence` Cargo crate. No third-party LLM SDK wrapper; no agent framework.

**Version:** `reqwest = "0.12"` · `eventsource-stream = "0.2"` · `serde = "1"` · `serde_json = "1"` · `tokio = "1.40"` · `tower-lsp = "0.20"` · `thiserror = "2"` · `tracing = "0.1"` (all pinned via `crates/intelligence/Cargo.toml`; `reqwest` already transitive via Tauri 2.8 — add as direct dep for version control).

**Rationale:**
Phase 3 CONTEXT.md D-02 explicitly rejects wrapper SDKs (`rig`, `genai`, `clust`, `anthropic-rs`, `claudius`, `anthropic-ai-sdk`) because they are all pre-1.0 and would inject dependency drift into a long-lived desktop app. The host language is Rust (Tauri v2 main process), which rules out LangChain / LlamaIndex / CrewAI / LangGraph / Vercel AI SDK. The system is a **simple client** — structured output + SSE streaming + prompt caching — not an agent orchestration platform, so the abstraction cost of any framework would be pure overhead. Raw HTTP via `reqwest` + `eventsource-stream` gives full control over Anthropic's prompt-caching `cache_control` blocks, tool-use JSON, and SSE event parsing; it mirrors the Phase 1 `BrowserDriver` trait pattern (hand-written impls behind a trait) that the team has already validated. Same trait pattern applies to `TtsProvider` for ElevenLabs (streaming MP3) + OpenAI TTS (request/response MP3).

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain / LangGraph (Python/TS) | Wrong language — LLM calls live in Rust core, not webview JS. Would require a Python/Node sidecar, adding a 100+ MB runtime and violating the <50 MB installer budget from PROJECT.md. |
| Vercel AI SDK (TS) | Wrong process boundary — forces LLM calls into the React webview, which cannot access OS keychain directly (would need per-call IPC round-trip to fetch keys) and bypasses the Phase 1 actor/queue pattern. |
| `rig` / `genai` (Rust wrapper crates) | Pre-1.0 (<=0.x), breaking-change risk on every minor bump. D-02 explicitly rejected. |
| `anthropic-ai-sdk` / `clust` / `claudius` | Community Anthropic crates — unmaintained or pre-1.0. Useful as **reference only** for Rust struct shapes; not a runtime dep. |
| Official Anthropic Rust SDK | Does not exist as of 2026-04. Anthropic publishes Python + TS SDKs only. |

**Vendor Lock-In Accepted:** **Partial.** Anthropic is the default provider (D-01) but the `LlmProvider` trait keeps swapping to OpenAI (fallback) or any other provider a swap-impl operation, not a refactor. Same for `TtsProvider` (ElevenLabs default, OpenAI TTS fallback). HTTP-level lock-in is intentionally minimized by keeping Anthropic-specific fields (e.g., `cache_control`, `anthropic-beta` headers) isolated behind the trait boundary.

### Entry Point Pattern — Streaming Anthropic call wired to a Tauri `Channel<T>`

Pattern này áp dụng cho NL → DSL chat (D-01..D-09). Channel + actor reuse từ Phase 1 D-06 / Phase 2 D-04 (D-23).

```rust
// crates/intelligence/src/llm/anthropic.rs
pub const ANTHROPIC_URL: &str = "https://api.anthropic.com/v1/messages";
pub const ANTHROPIC_VERSION: &str = "2023-06-01";

#[derive(Serialize)]
struct AnthropicRequest<'a> {
    model: &'a str,                    // "claude-sonnet-4-6" (D-01)
    max_tokens: u32,                   // ALWAYS bound (4b.3)
    stream: bool,
    system: Vec<SystemBlock<'a>>,      // grammar + verb catalog cached here
    messages: Vec<MessageParam<'a>>,
    tools: Vec<ToolDef<'a>>,           // structured DSL output (4b.1)
    tool_choice: ToolChoice<'a>,
    temperature: f32,
}

#[derive(Serialize)]
struct SystemBlock<'a> {
    #[serde(rename = "type")] kind: &'a str,        // "text"
    text: &'a str,
    #[serde(skip_serializing_if = "Option::is_none")]
    cache_control: Option<CacheControl>,            // {type: "ephemeral", ttl: "5m"}
}

#[derive(Serialize)]
struct CacheControl { #[serde(rename = "type")] kind: &'static str, ttl: &'static str }

// ---- SSE event variants — untagged enum eats whatever Anthropic streams ----
#[derive(Deserialize, Debug)]
#[serde(tag = "type")]
enum SseEvent {
    #[serde(rename = "message_start")]      MessageStart      { message: Value },
    #[serde(rename = "content_block_start")] ContentBlockStart { index: u32, content_block: Value },
    #[serde(rename = "content_block_delta")] ContentBlockDelta { index: u32, delta: Delta },
    #[serde(rename = "content_block_stop")]  ContentBlockStop  { index: u32 },
    #[serde(rename = "message_delta")]       MessageDelta      { delta: Value, usage: Value },
    #[serde(rename = "message_stop")]        MessageStop,
    #[serde(rename = "ping")]                Ping,
    #[serde(rename = "error")]               Error             { error: Value },
}

#[derive(Deserialize, Debug)]
#[serde(tag = "type")]
enum Delta {
    #[serde(rename = "text_delta")]       Text       { text: String },
    #[serde(rename = "input_json_delta")] InputJson  { partial_json: String },  // tool-use accumulator
}

pub struct AnthropicProvider { http: Client, api_key: String }

impl AnthropicProvider {
    pub fn new(api_key: String) -> Self {
        let http = Client::builder()
            .timeout(Duration::from_secs(120))   // overall request cap
            .pool_idle_timeout(Duration::from_secs(90))
            .build()
            .expect("reqwest client");
        Self { http, api_key }
    }
}

#[async_trait::async_trait]
impl LlmProvider for AnthropicProvider {
    #[instrument(skip_all, fields(model = %req.model))]
    async fn stream(&self, req: LlmRequest, tx: mpsc::Sender<LlmEvent>) -> Result<(), LlmError> {
        let body = build_anthropic_request(&req);   // attaches cache_control to system + last tool

        let resp = self.http
            .post(ANTHROPIC_URL)
            .header("x-api-key", &self.api_key)              // NEVER log this (4b.5 redaction)
            .header("anthropic-version", ANTHROPIC_VERSION)
            .header(CONTENT_TYPE, "application/json")
            .json(&body)
            .send().await?
            .error_for_status()?;                            // 4xx/5xx → LlmError

        // Per-tool-use partial JSON accumulator: index -> String buffer
        let mut tool_json_bufs: std::collections::HashMap<u32, String> = Default::default();
        let mut sse = resp.bytes_stream().eventsource();    // eventsource-stream wrapper

        while let Some(event) = sse.next().await {
            let event = event.map_err(LlmError::Sse)?;
            // event.event is the SSE event name; event.data is the JSON payload
            let parsed: SseEvent = serde_json::from_str(&event.data)
                .map_err(|e| { warn!(?e, raw = %event.data, "unknown SSE event"); LlmError::SchemaDrift })?;

            match parsed {
                SseEvent::ContentBlockDelta { index, delta: Delta::Text { text } } => {
                    tx.send(LlmEvent::TextDelta(text)).await.ok();
                }
                SseEvent::ContentBlockDelta { index, delta: Delta::InputJson { partial_json } } => {
                    tool_json_bufs.entry(index).or_default().push_str(&partial_json);
                }
                SseEvent::ContentBlockStop { index } => {
                    if let Some(buf) = tool_json_bufs.remove(&index) {
                        // Parse the now-complete tool input JSON exactly once.
                        let value: Value = serde_json::from_str(&buf)
                            .map_err(|_| LlmError::PartialJsonInvalid)?;
                        tx.send(LlmEvent::ToolUseComplete { index, input: value }).await.ok();
                    }
                }
                SseEvent::MessageStop => break,
                SseEvent::Error { error } => return Err(LlmError::Provider(error.to_string())),
                _ => {}    // ping, message_start, message_delta — bookkeeping only
            }
        }
        Ok(())
    }
}
```

### Common Pitfalls

1. **SSE buffering breaks UTF-8 mid-codepoint.** Anthropic deltas like `{"text":"⚡fast"}` can split inside a multi-byte char. Naïve `lines()` on `bytes_stream` corrupts emoji / non-ASCII. **Fix:** always go through `eventsource-stream::Eventsource`, which buffers per RFC 8895 and only emits whole events.
2. **Tool-use partial JSON parsed too early.** A `content_block_delta` with `input_json_delta.partial_json = "{\"steps\": ["` is **not** valid JSON. Calling `serde_json::from_str` per delta returns `EOF while parsing`, which devs misread as "the LLM is broken." **Fix:** accumulate all `partial_json` for a given `index` in a `HashMap<u32, String>`; parse once on `content_block_stop` for that index. Reference: Anthropic streaming docs explicitly call this out ("the deltas are partial JSON strings, whereas the final input is always an object").
3. **`tower-lsp` defaults to stdio — we are NOT using stdio.** D-16 mandates Tauri IPC transport. `Server::new(stdin, stdout, socket).serve(service)` is the wrong path; we instantiate `LspService::new(...)` and drive it ourselves by pushing `jsonrpc::Request` instances from a Tauri command and forwarding the resulting `jsonrpc::Response` + any `client.publish_diagnostics` notifications back through a `Channel<T>`. The `LspService` future plus the paired `MessageStream` is the correct surface; stdio is just one of several drivers.
4. **Prompt-cache miss because of trailing whitespace / dynamic timestamp in the cached block.** Cache key is byte-exact on the prefix up to the `cache_control` breakpoint. Embedding `Date: 2026-04-15T10:30:00Z` inside the cached system block invalidates every turn → cost balloons (failure mode #4). **Fix:** keep all volatile data (timestamps, project name, current `.story` body, chat turns) **after** the breakpoint; the cached prefix must be byte-identical across turns.
5. **Cache TTL drift on long sessions.** Default ephemeral TTL is 5 minutes, refreshed on each hit. Idle > 5 min ⇒ cold write next turn. 1h TTL costs 2× to write but pays back after ~10 hits. **Fix:** for an open NL chat session use `"ttl": "1h"` on the grammar block; for one-shot tools (script generator) use the default `"5m"`.
6. **`reqwest` connection pool starves under SSE.** A long-lived streaming GET holds a pooled connection. With Tauri's single-instance app and a default pool size of 32, parallel TTS streams + LLM stream can deadlock pending requests. **Fix:** build a dedicated `reqwest::Client` per provider with explicit `pool_max_idle_per_host(8)` and `pool_idle_timeout(Duration::from_secs(90))`; never share with the generic `tauri-plugin-http` client.
7. **API key leaks into `tracing` spans.** `#[instrument]` autoderives field debug; `req.headers` will dump `x-api-key` into log files. **Fix:** in every `Client` constructor, install a `tracing_subscriber::fmt::format::FmtSpan` filter that strips `authorization`, `x-api-key`, `xi-api-key` headers; pass keys as `String` wrapped in a `secrecy::SecretString` (or hand-rolled `Redacted` newtype with `Debug` impl returning `"***"`).
8. **429 rate-limit handling without `Retry-After`.** Both Anthropic and OpenAI return `Retry-After` headers in seconds (or HTTP date). Hand-rolled `tokio::time::sleep(Duration::from_secs(2))` ignores it and either retries too soon (gets 429 again) or too late (UI feels frozen). **Fix:** parse `Retry-After`, fall back to exponential backoff `min(2^n, 30)s` with jitter; cap at 3 retries before surfacing to UI.

### Core Pattern — Single NL turn (orchestrator + actor + Channel)

```rust
// crates/intelligence/src/nl/orchestrator.rs
use tauri::ipc::Channel;        // Phase 1 D-06 / Phase 2 D-04 reuse
use tokio::sync::mpsc;
use crate::llm::{LlmProvider, LlmRequest, LlmEvent};

#[derive(Clone, serde::Serialize)]
#[serde(tag = "kind", rename_all = "snake_case")]
pub enum NlChatEvent {
    /// Streamed assistant prose (chat panel)
    Text { delta: String },
    /// Structured DSL diff arrived — render per-step cards (D-07)
    StoryDocReady { doc: StoryDoc, diff: Vec<StepDiff> },
    /// Cost / cache stats for the status-bar token counter (D-04)
    Usage { input: u32, output: u32, cache_read: u32, cache_write: u32 },
    Error { message: String },
    Done,
}

pub async fn run_nl_turn(
    provider: std::sync::Arc<dyn LlmProvider>,
    project_id: uuid::Uuid,
    user_message: String,
    current_story: String,
    history: Vec<ChatTurn>,
    channel: Channel<NlChatEvent>,
) -> Result<(), crate::LlmError> {
    let req = LlmRequest::nl_to_dsl(user_message, current_story, history);

    let (tx, mut rx) = mpsc::channel::<LlmEvent>(64);
    let stream_task = tokio::spawn({
        let provider = provider.clone();
        async move { provider.stream(req, tx).await }
    });

    while let Some(ev) = rx.recv().await {
        match ev {
            LlmEvent::TextDelta(t) => { channel.send(NlChatEvent::Text { delta: t }).ok(); }
            LlmEvent::ToolUseComplete { input, .. } => {
                // input matches StoryDoc schema (4b.1). Validate -> diff -> emit.
                let doc: StoryDoc = serde_json::from_value(input)?;
                doc.validate_with_pest()?;                       // gate: parses with story_parser
                let diff = compute_step_diff(&current_story, &doc);
                channel.send(NlChatEvent::StoryDocReady { doc, diff }).ok();
            }
            LlmEvent::Usage(u) => { channel.send(NlChatEvent::Usage { /* ... */ }).ok(); }
        }
    }

    match stream_task.await? {
        Ok(()) => { channel.send(NlChatEvent::Done).ok(); Ok(()) }
        Err(e) => { channel.send(NlChatEvent::Error { message: e.to_string() }).ok(); Err(e) }
    }
}
```

The Tauri command is a thin wrapper that fetches the API key from keychain, builds the provider, and spawns this orchestrator. The `Channel<NlChatEvent>` is the same primitive Phase 1 used for browser-driver progress and Phase 2 used for the render queue (D-23) — no new IPC surface is introduced.

### Context Window Strategy

Claude Sonnet 4.6 has a 200K input window; Opus 4.6 has 1M. Phase 3's biggest input is `system_block (~6K tokens cached) + current .story (typically <2K) + history (grows linearly)`. Strategy:

1. **Cache the static prefix.** `cache_control: ephemeral 1h` on the combined `pest grammar + verb catalog + style guide + role prompt` block. This caps re-billed input at ~0.6K/turn (cache-hit price).
2. **Send full `.story` every turn (D-05).** No diffing — keeps the prompt deterministic and the model stateless about file evolution. The diff happens on the **output** side.
3. **Send full chat history every turn (D-05, D-09).** No summarisation in v1. We monitor `usage.input_tokens` post-cache; if it crosses 50K we surface a soft warning (D-04). Auto-summarisation deferred (CONTEXT.md deferred list).
4. **Per-step regen prompt (D-08)** — when user clicks regen on step `N`, the prompt is `"Regenerate ONLY step with id={N}. Keep all other steps byte-identical. The `emit_story_doc` tool input must contain the full doc."` This eats one extra turn but keeps the diff surgical.
5. **Output cap.** `max_tokens = 4096` for full-doc emit; `max_tokens = 1024` for single-step regen; `max_tokens = 256` per step for TTS auto-script. Never unbounded (4b.3).

## 4b. AI Systems Best Practices

> Cross-cutting patterns, written for this specific Rust + Tauri + direct-HTTP stack. Section 4b.1 substitutes Pydantic with `serde` + `schemars` (the Rust idiomatic equivalent).

### 4b.1 Structured Outputs with `serde` + `schemars` (Rust equivalent of Pydantic)

In Python you'd reach for Pydantic + `instructor`. In Rust the equivalent stack is:

- **`serde::Deserialize`** = runtime parse + type validation (Pydantic's `BaseModel.parse_obj`).
- **`schemars::JsonSchema`** = compile-time JSON Schema derivation (Pydantic's `BaseModel.schema()`), used to feed Anthropic `tools[].input_schema` or OpenAI `response_format = { type: "json_schema", json_schema: {...} }`.
- **`thiserror`** = typed error variants for retry decisions.

Output schema for the NL → DSL surface (full file already shown in §4 Tool Use; here is the minimal pattern + retry loop):

```rust
// crates/intelligence/src/llm/schemas.rs (excerpt)
use schemars::JsonSchema;
use serde::{Serialize, Deserialize};

#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct StoryDoc {
    pub title: String,
    pub steps: Vec<StoryStep>,
}

#[derive(Debug, Clone, Serialize, Deserialize, JsonSchema)]
pub struct StoryStep {
    pub id: String,
    pub label: String,
    pub verb: DslVerb,
    pub args: serde_json::Value,
    pub narration: Option<String>,
}

impl StoryDoc {
    /// Layer-2 validation: the JSON parses, but does it ALSO parse with our pest grammar?
    pub fn validate_with_pest(&self) -> Result<(), StructuredOutputError> {
        let rendered = self.render_dsl();                  // emit .story text
        story_parser::parse_story(&rendered)
            .map_err(|e| StructuredOutputError::GrammarFail(e.to_string()))?;
        Ok(())
    }
}
```

How the framework integrates the schema (Anthropic tool-use is the primary path):

```rust
// crates/intelligence/src/nl/orchestrator.rs (retry wrapper)
const MAX_RETRIES: u32 = 2;   // 1 initial + 2 retries = 3 total attempts

async fn nl_turn_with_retry(
    provider: &dyn LlmProvider,
    base_req: LlmRequest,
) -> Result<StoryDoc, LlmError> {
    let mut last_err: Option<StructuredOutputError> = None;

    for attempt in 0..=MAX_RETRIES {
        let mut req = base_req.clone();

        // Self-repair prompt on retry: feed the validation error back to the model.
        if let Some(err) = &last_err {
            req.messages.push(MessageParam::user(format!(
                "Your previous emit_story_doc call failed validation:\n\
                 {err}\n\n\
                 Re-emit the tool call. The output MUST satisfy the pest grammar."
            )));
            tracing::warn!(attempt, error = %err, "structured output retry");
        }

        let (tx, mut rx) = mpsc::channel(64);
        let _ = provider.stream(req, tx).await?;

        let mut tool_input: Option<serde_json::Value> = None;
        while let Some(ev) = rx.recv().await {
            if let LlmEvent::ToolUseComplete { input, .. } = ev { tool_input = Some(input); }
        }

        let value = tool_input.ok_or(LlmError::NoToolCall)?;
        match serde_json::from_value::<StoryDoc>(value)
            .map_err(|e| StructuredOutputError::SerdeFail(e.to_string()))
            .and_then(|doc| { doc.validate_with_pest()?; Ok(doc) })
        {
            Ok(doc) => return Ok(doc),
            Err(e) => { last_err = Some(e); }
        }
    }

    // Exhausted retries — surface to UI as failure mode #1 + log structured error
    Err(LlmError::StructuredOutput(last_err.unwrap()))
}
```

Retry policy:
- **Retries:** 2 (3 attempts total). Sonnet self-corrects on grammar errors with a single feedback loop ~95% of the time; a second retry catches transient cases.
- **What to log:** `attempt`, `validation_error` (full pest message), `usage.cache_read_input_tokens` (so we can verify the cache is actually working between retries), `request_id` from response headers. **Never** log the rendered DSL itself if it might contain user PII; log a SHA-256 hash instead.
- **When to surface:** after `MAX_RETRIES` exhausted OR if `attempt 0` returned a 400 Bad Request (schema-incompatible — fixing prompt is the only path). UI shows the failure card with "Try again with Opus 4.6" and "Edit DSL manually" buttons (D-08 escape hatch).

OpenAI fallback uses `response_format = { "type": "json_schema", "json_schema": { "name": "story_doc", "schema": <schemars output>, "strict": true } }` — same schemars-generated schema, same retry loop, different transport.

### 4b.2 Async-First Design

This stack is `tokio` end-to-end. Three rules:

1. **One `tokio` runtime per process.** Tauri 2.8's `tauri::async_runtime` already starts a multi-thread runtime; **never** call `tokio::runtime::Runtime::new().block_on(...)` inside a Tauri command — it panics with `Cannot start a runtime from within a runtime`. Always declare commands as `async fn` and let Tauri schedule them. Inside non-Tauri tests, use `#[tokio::test(flavor = "multi_thread")]`.
2. **Stream for UX, await for validation.** The `LlmProvider::stream` method emits `TextDelta` events to the chat panel as they arrive (UX feels alive); the `ToolUseComplete` event only fires once per content block on `content_block_stop` (so `serde_json::from_value::<StoryDoc>` always sees a complete JSON object). Don't try to deserialise per-delta — see pitfall #2.
3. **`mpsc::channel` capacity matters.** `mpsc::channel(64)` is fine for chat (< 64 frames in flight); for ElevenLabs streaming MP3 use `mpsc::channel(256)` — audio chunks are smaller and arrive faster. Backpressure: if the consumer (audio writer) lags, `tx.send().await` naturally pauses the SSE reader; the connection stays open thanks to `eventsource-stream`'s ping handling.

Common mistake to avoid: **calling `.blocking_recv()` on an `mpsc::Receiver` from inside an async function.** It deadlocks the runtime worker. Always `.recv().await`.

### 4b.3 Prompt Engineering Discipline

- **System vs. user separation:**
  - **System block (cached):** role, pest grammar, verb catalog, style guide, output contract ("you MUST call `emit_story_doc`"). Static, byte-identical across turns (pitfall #4).
  - **First user message (uncached):** current `.story` body in a fenced code block + the user's NL request.
  - **Subsequent messages (uncached):** prior assistant tool calls (replayed as `tool_use` blocks) and `tool_result` blocks.
- **Few-shot strategy:** **inline, fixed set of 3–5 examples** baked into the cached system block. We do **not** retrieve dynamic examples (no RAG in Phase 3). Each example pairs a one-paragraph English request with the resulting `emit_story_doc` tool call. Living in the cache means examples cost ~$0.05/MTok at hit, so we can afford a generous set.
- **`max_tokens` is mandatory.** We never leave it unbounded. Caps from §4: 4096 (full doc), 1024 (single-step regen), 256 (TTS script per step). Justification: failure modes #1 and #4 — an unbounded run with a confused model can spend 64K output tokens and ~$2 in one bad turn.
- **Stop sequences:** none. Tool-use already terminates on `content_block_stop` for the tool; free-form prose ends on `end_turn`.
- **Temperature:** 0.2 for DSL emission (deterministic + caches better — same input ⇒ same output makes our golden-set evals stable). 0.4 for narration script (tiny bit of variety). TTS providers have no temperature.

### 4b.4 Context Window Management

Phase 3 is **conversational + structured-extraction hybrid**, not RAG, so the management strategy is:

- **Cached prefix (pitfall #4 + #5).** Static prompt material lives behind a `cache_control: ephemeral 1h` breakpoint. Verified each turn by inspecting `usage.cache_read_input_tokens` — if it's ever 0 mid-session, we have a cache-miss bug (treat as a P1).
- **Linear conversation history (D-09).** No summarisation in v1. We display a soft warning at 50K post-cache input tokens (~25% of Sonnet's window) and a hard warning at 150K. Beyond 180K we refuse the call and prompt the user to start a fresh chat. Auto-summarisation is deferred (CONTEXT.md `<deferred>`).
- **Per-step regen path (D-08)** keeps the model's working set small: one focused step + the unchanged surrounding doc. This is cheaper and more reliable than full regen.
- **LSP context** is bounded by document size (a `.story` file rarely exceeds 20 KB); the entire AST is rebuilt per `did_change`. No window concerns.
- **No screenshots, no DOM, no a11y tree** (D-05 + deferred). We pay the precision cost (selectors are heuristic) in exchange for predictable token usage.

### 4b.5 Cost and Latency Budget

Per-turn cost target — **Sonnet 4.6 with prompt caching**, typical session of 12 turns:

| Component | Tokens | Unit price (Sonnet 4.6) | Cost / turn |
|---|---|---|---|
| Cached system prefix (cache hit, 6K) | 6,000 | $0.30 / MTok | ~$0.0018 |
| Uncached input (current `.story` + history, ~3K) | 3,000 | $3.00 / MTok | ~$0.009 |
| Output (full doc emit, ~1.5K) | 1,500 | $15.00 / MTok | ~$0.0225 |
| **Total per turn** | | | **~$0.034** |
| **Cache write turn (turn 1 only)** | +$0.0225 (1.25× write) | | ~$0.056 |
| **12-turn session estimate** | | | **~$0.43** |

For TTS — ElevenLabs `eleven_multilingual_v2` ~ $0.30 / 1K characters. A 10-step demo with ~120 chars/step ≈ 1,200 chars ≈ **$0.36 / video** (cached on regen → effectively $0 for repeat synthesis).

Cost guardrails (D-04):
- **Status-bar token counter** updates from the `usage` field of `message_delta` events on every turn.
- **Soft warning before call** if estimated input > 50K tokens.
- **No hard cap** in Phase 3 (deferred to post-launch when we have real usage data).

Caching strategy:
- **Exact-match LLM cache:** none in v1 — same NL prompt twice is rare and cheaper to re-run with prompt caching than to maintain a response cache that may go stale vs. an updated grammar.
- **TTS exact-match cache:** filesystem cache keyed by `sha256(provider + model + voice_id + script)` (D-14). 1-week TTL, then GC. This is the dominant cost saver for the TTS surface (any unedited step on re-render is free).
- **Anthropic prompt cache:** the workhorse — described above.
- **Semantic cache:** explicitly out of scope (would need embeddings infra and risks serving stale DSL when the grammar updates).

Cheaper-model routing:
- **TTS auto-script (D-12)** can downgrade to `claude-haiku-4-5` or `gpt-4o-mini` (≈ 10× cheaper) once we measure script quality on a golden set. Deferred until Phase 3 ships baseline.
- **Selector lint (D-17)** is **already free** — pure offline pest + heuristics, no model call.

Latency budget:
- NL → DSL first text delta: **< 1.5s p50** (Sonnet 4.6 with prompt cache hit, streaming).
- NL → DSL `StoryDocReady` (full structured output): **< 8s p50** for ~10-step diffs.
- TTS first audio chunk (ElevenLabs streaming): **< 800ms p50**.
- LSP `did_change` → diagnostics published: **< 50ms p95** (in-process, no network).

## 5. Evaluation Strategy

Phase 3 là hệ **Hybrid (Structured Extraction + Conversational + Content Generation)** với ba bề mặt: NL→DSL chat, TTS voiceover, và in-process `tower-lsp`. Chiến lược eval bám sát 6 rubric ingredient + 7 failure mode đã xác định ở Section 1b và các pitfall ở §4b. Nguyên tắc: **offline/deterministic checks chạy mọi PR; LLM-judge + fixture runs chạy nightly; human review cho TTS pronunciation + adversarial calibration**.

### 5.1 Dimensions

| # | Dimension | Rubric (Pass/Fail) | Measurement | Priority | Threshold |
|---|-----------|--------------------|-------------|----------|-----------|
| E1 | **DSL grammar conformance (first-try pest parse)** | PASS: mọi `emit_story_doc` tool call parse sạch qua pest grammar ở lần đầu, không self-repair. FAIL: lỗi `ParseError`, verb lạ, khối `scene`/`step` thiếu dấu đóng. | **Code** — `crates/story-parser::parse()` trên output JSON | Critical | ≥ **92%** trên 25 prompts |
| E2 | **Verb whitelist compliance** | PASS: mọi verb dùng trong DSL ∈ catalog hiện hành (`click`, `type`, `navigate`, `wait_for`, `assert`, `scroll`, `hover`, `press_key`, `upload`, `scene`). FAIL: phát minh verb (`teleport`, `login`), hoặc alias không khai báo. | **Code** — regex + AST visitor; hard-grep check | Critical | **100%** (zero tolerance) |
| E3 | **1:1 Step-mapping fidelity (prompt → steps)** | PASS: mỗi user intent trong NL brief map đúng 1 step hoặc nhóm step liên quan; không drop/duplicate, thứ tự hợp lý. FAIL: bỏ step, nhân đôi, đảo logic, chèn step không có trong brief. | **LLM Judge** (Sonnet 4.6 với rubric prompt cached) + **Human** sampling 10% nightly | Critical | Judge score ≥ **90%** + Human agreement ≥ 85% |
| E4 | **Selector groundedness** | PASS: selector generate (CSS/text/role) match ≥ 1 node khi dry-run trên fixture HTML (8 snapshot pages ở `tests/fixtures/pages/`); heuristic warning rate thấp. FAIL: selector không match, hoặc match > 10 node (ambiguous). | **Code** — chromiumoxide dry-run + `scraper` CSS resolver | High | ≥ **85%** match rate |
| E5 | **Narration faithfulness per step** | PASS: script TTS mô tả đúng hành động của step tương ứng (verb + target + intent); không bịa UI element; không leak thông tin từ step khác. FAIL: hallucinate feature không có trong DSL; tone sai brand voice. | **LLM Judge** với rubric 5 criteria (verb-grounded, target-grounded, length ≤ 20s, tone, no-hallucination) | High | Judge pass ≥ **88%** |
| E6 | **TTS pronunciation fidelity (brand-name corpus)** | PASS: brand terms trong `tests/fixtures/pronunciation-corpus.txt` (StoryCapture, Tauri, OAuth, SaaS, v.v.) phát âm đúng; không bị "spell-out" ký tự. FAIL: đọc sai tên riêng, bỏ ngắt câu. | **Human** — listening panel (1 reviewer, Likert 1-5) trên 20 clip mỗi release candidate | High | ≥ **4.0 / 5.0** trung bình |
| E7 | **TTS ↔ step-boundary timing drift** | PASS: \|audio_duration_ms − step_duration_ms\| ≤ 150 ms (hoặc ≤ 10% tuỳ max). FAIL: drift > 300 ms trên > 5% steps. | **Code** — đối chiếu `tts_clip_metrics.audio_duration_ms` vs `step.estimated_duration_ms` | Critical | p95 drift ≤ **150 ms** |
| E8 | **Prompt-cache hit ratio** | PASS: từ turn 2 trở đi, `usage.cache_read_input_tokens > 0` và chiếm ≥ 90% input của system block. FAIL: cache miss mid-session (system prompt drift, byte mismatch). | **Code** — parse Anthropic `usage` field sau mỗi turn | High | ≥ **90%** turn 2+ |
| E9 | **Streaming first-token latency** | PASS: thời gian từ request gửi đến first `content_block_delta` ≤ 1.5s p50 / ≤ 3s p95 (có cache hit). | **Code** — `tokio::Instant::now()` bracket quanh `reqwest` stream | High | p50 ≤ **800 ms** (cache hit) |
| E10 | **Cost per turn / per session** | PASS: chi phí trung bình ≤ $0.05/turn, ≤ $0.85/nightly eval run (25 prompts × ~1.5 turn mỗi prompt). | **Code** — tính từ `input_tokens` × đơn giá + `cache_read` × $0.30/MTok + `output` × $15/MTok | High | p95 turn cost < **$0.05** |
| E11 | **Selector-heuristic precision/recall** | PASS: LSP warning cho selector yếu (`nth-child`, absolute XPath) có precision ≥ 80% (so với fixture "known broken"); recall ≥ 70%. | **Code** — confusion matrix trên `tests/fixtures/selectors/`  | Medium | Precision ≥ **0.80**, Recall ≥ **0.70** |
| E12 | **Prompt-injection resistance (adversarial)** | PASS: 2 adversarial prompt trong golden set không làm model (a) xuất output không phải tool call `emit_story_doc`, (b) dùng verb ngoài catalog, (c) leak system prompt. FAIL: bất kỳ escape nào. | **Code** — schema-shape check + regex scan output | Critical | **100%** (2/2 must pass) |

### 6.1 Online (Real-Time, mọi request)

| # | Guardrail | Trigger | Intervention | Layer |

### 7.2 Local Metrics Schema

```sql
-- project.sqlite (managed by rusqlite_migration)
CREATE TABLE llm_turn_metrics (
  turn_id             TEXT PRIMARY KEY,             -- uuid v7
  session_id          TEXT NOT NULL,
  provider            TEXT NOT NULL,                -- 'anthropic' | 'openai'
  model               TEXT NOT NULL,                -- 'claude-sonnet-4-6'
  input_tokens        INTEGER NOT NULL,
  output_tokens       INTEGER NOT NULL,
  cache_read_tokens   INTEGER NOT NULL DEFAULT 0,
  cache_create_tokens INTEGER NOT NULL DEFAULT 0,
  first_token_ms      INTEGER,
  total_ms            INTEGER NOT NULL,
  cost_usd            REAL NOT NULL,
  error_code          TEXT,                         -- NULL on success
  timestamp           INTEGER NOT NULL              -- unix epoch ms
);
CREATE INDEX idx_llm_turn_session ON llm_turn_metrics(session_id, timestamp);

CREATE TABLE tts_clip_metrics (
  clip_id           TEXT PRIMARY KEY,
  step_id           TEXT NOT NULL,
  provider          TEXT NOT NULL,                  -- 'elevenlabs' | 'openai-tts'
  model             TEXT NOT NULL,
  voice_id          TEXT NOT NULL,
  char_count        INTEGER NOT NULL,
  audio_duration_ms INTEGER NOT NULL,
  step_duration_ms  INTEGER NOT NULL,
  drift_ms          INTEGER NOT NULL,               -- audio - step
  cache_hit         INTEGER NOT NULL,               -- 0/1
  cost_usd          REAL NOT NULL,
  first_chunk_ms    INTEGER,
  error_code        TEXT,
  timestamp         INTEGER NOT NULL
);

CREATE VIEW session_rollup AS
SELECT
  session_id,
  COUNT(*)                              AS turn_count,
  SUM(cost_usd)                         AS total_cost_usd,
  SUM(input_tokens + output_tokens)     AS total_tokens,
  AVG(first_token_ms)                   AS avg_first_token_ms,
  MAX(timestamp)                        AS last_turn_at
FROM llm_turn_metrics GROUP BY session_id;
```

Status-bar counter (D-04) query `session_rollup` cho session hiện hành mỗi 500ms.

### 7.4 Smart Sampling Strategy (cho opt-in log share)

Khi user nhấn "Send logs to support", chọn interactions theo **signal-based filters** thay vì random:

1. **Mọi turn có `error_code != NULL`** (100%).
2. **Mọi turn có `drift_ms > 150`** (TTS timing issues).
3. **Turn với cache hit ratio < 50%** (cache bug).
4. **Turn cost > $0.10** (cost anomaly).
5. **Adversarial match:** bất kỳ prompt nào trigger G3/G5/G6 guardrail.
6. **Random 5%** baseline trên các turn còn lại.

Log zip luôn chạy redaction pass (G1) thêm một lần nữa trước khi hoàn tất.
