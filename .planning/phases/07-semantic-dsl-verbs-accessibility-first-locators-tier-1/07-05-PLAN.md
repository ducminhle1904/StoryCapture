# Phase 07-05: Author-time selector validator + hover-preview

## Purpose

Turn Phase 7's DSL grammar + locator engine into an author-time **typechecker**: validate every selector the user writes against a cached DOM snapshot, surface the result inline in the editor, and preview the matched element on hover. This is the reverse direction of Phase 7's picker (picker = click element → DSL; validator = DSL → highlight element).

## Design summary

- **Snapshot cache.** A dedicated author-time Playwright page (separate from any recording session) loads each URL referenced by `Navigate` steps. For each URL we persist `(domHash, innerHTML, screenshot.png)` in the project's `.story.snapshots/` dir keyed by SHA-256(url). Snapshots are single-shot; the user's "Refresh snapshot" button re-captures.
- **Locator engine reuse.** `SmartSelector::validate_against_dom` is a new method on the existing selector crate — runs the same ranked strategy list the sidecar uses, but against a detached DOM string (via `lol_html` or `kuchiki`). Returns `unique | fuzzy(count) | none` plus the first matched element's bounding box.
- **CodeMirror integration.** Gutter marker per step line + hover decoration. Hovering a step sends the parsed target + URL context to the validator, which returns `(chip_state, bbox?)`. Preview panel then draws the snapshot screenshot with the bbox overlay.
- **Phase 7 synergy.** When validation returns `fuzzy(N)` and the user clicks "Promote unique alternative to fallback", we write that locator into `.story.targets.json`'s fallbacks — exactly the self-healing shape 07-04c defines. Zero new file format.

## Out of scope

- Running the DSL to produce snapshots (author-time executor is Phase 10).
- Auto-refreshing snapshots when the target site changes (manual for now).
- Rendering the author-time Playwright browser live — that's Phase 9-04.

## Risk flags

- **DOM drift:** cached DOM → stale if site changes. Mitigation: snapshot timestamp in the chip tooltip; "Refresh" button visible when chip is red/yellow.
- **Performance on long stories:** validator runs per step; debounce + memoize on `(step_hash, dom_hash)`.
- **Phase 7 locator engine is Playwright-side (Node):** `validate_against_dom` needs a Rust implementation of the same ranking — duplicate logic risk. Mitigation: share grammar via `story-parser`; port only the deterministic strategies (CSS, TestId, label-for) to Rust; YELLOW-degrade the fuzzy-text strategies that require live DOM.
