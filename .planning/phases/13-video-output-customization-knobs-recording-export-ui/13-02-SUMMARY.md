# Phase 13 Plan 02: Desktop Infrastructure Summary

## One-liner
Wired tauri-plugin-store and scaffolded six Base UI primitives plus two bespoke field wrappers so Plans 13-03/04/05 can import them without further infra work.

## Objective
Land the scaffolding Phase 13 needs that has nothing to do with output-prefs business logic: register tauri-plugin-store (first production use), expand capabilities to permit store + per-project fs, and scaffold the shadcn Base UI primitives Output Preferences will consume.

## Execution

### Task 1 тАФ tauri-plugin-store + capabilities
- `npm view @tauri-apps/plugin-store version` тЖТ 2.4.2; `cargo search` тЖТ 2.4.2. Both match CLAUDE.md stack table validated band (2.4.x).
- Added `tauri-plugin-store = "2.4"` to Cargo.toml (alphabetical among tauri-plugin-*); added `@tauri-apps/plugin-store ^2.4.0` to package.json.
- Registered `tauri_plugin_store::Builder::default().build()` in lib.rs plugin chain immediately after `tauri_plugin_fs::init()`.
- Appended to capabilities/default.json: `store:default`, plus fs read-text-file/write-text-file scoped to `$APPDATA/**` and `$APPCONFIG/**`. Existing `$HOME/**` scope already covers arbitrary user project dirs (Plan 13-03 writes `<project>/.storycapture/output.json` under user's home by default).
- Verified: `cargo build -p storycapture` тЖТ green (with sidecar placeholders installed via existing `scripts/dev/install-sidecar-placeholders.sh`). JSON parses via `node -e JSON.parse(...)`.
- Commit: `c9a950c`

### Task 2 тАФ 6 shadcn primitives + 2 bespoke wrappers
- Ran `pnpm dlx shadcn@latest add accordion toggle-group radio-group slider input label`. CLI initially rejected `components.json` because shadcn 4.3 tightened the schema: `registries` is now an object (keyed by `^@`), not an array, and `style` must be one of the enumerated values (we had `new-york`, need `base-vega` for Base UI variant). Fixed components.json first, then CLI generated 7 files (including incidental `toggle.tsx`).
- Discovered CLI emitted `@base-ui/react/*` imports тАФ that package does not exist in node_modules; project ships `@base-ui-components/react`. Per CLAUDE.md "no workarounds": rewrote each primitive from scratch in the select.tsx rhythm (forwardRef + cn + token classes + displayName), using the correct Base UI parts API (e.g. `BaseAccordion.Root/Item/Header/Trigger/Panel`, `BaseSlider.Root/Control/Track/Indicator/Thumb`, `BaseRadio.Root/Indicator`).
- Deleted `toggle.tsx` тАФ not requested in the plan and duplicated button CVA variants.
- Token sweep: every color/radius uses `var(--color-*)` / `var(--radius-*)` tokens from `packages/ui/src/tokens.css`.
- Authored `color-field.tsx` (native `<input type=color>` + synced lowercase hex text, aria-invalid when not matching `/^#[0-9a-f]{6}$/`) and `number-field.tsx` (Input wrapper with numeric parse + min/max/step + aria-invalid wiring).
- Fix (Rule 2 тАФ critical a11y): `Label` now requires `htmlFor: string` as a typed prop so biome `a11y/noLabelWithoutControl` is satisfied at call sites. Shadcn's default loose signature would force every caller to remember to pass it; making it required is the root-cause fix.
- `pnpm typecheck` тЖТ green; `biome check` on the 8 new files тЖТ 0 errors.
- Incidental: `biome check --write` also normalized import order on pre-existing `button.tsx`, `dialog-motion.ts`, `select.tsx` (lineWidth 100 + organizeImports). Included in the same commit to keep working tree clean.
- Commit: `d72a88e`

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 тАФ Blocking] components.json schema mismatch vs shadcn 4.3**
- Found during: Task 2 Step A
- Issue: shadcn CLI (v4.3.0) rejected `components.json` with "Invalid configuration". Root cause: `registries: ["base-ui"]` (array) is invalid in shadcn 4.x schema (expects object keyed by `^@...`); `style: "new-york"` was also not the Base UI variant тАФ should be `base-vega`.
- Fix: Switched `style` тЖТ `base-vega`, removed `registries` key entirely. CLI now runs clean and uses the Base UI registry implicit in the `base-vega` style.
- Files: apps/desktop/components.json
- Commit: `d72a88e`

**2. [Rule 1 тАФ Bug] shadcn 4.3 base-vega emits wrong package imports**
- Found during: Task 2 Step B
- Issue: Generated files imported from `@base-ui/react/*`; that package does not exist in the workspace. The committed package is `@base-ui-components/react` (confirmed in select.tsx and node_modules). CLAUDE.md "no workarounds" forbids monkey-patching the CLI output.
- Fix: Rewrote every primitive from first principles using the same rhythm as select.tsx (the existing canonical Base UI shell). Kept shadcn naming conventions for exports.
- Files: all 6 scaffolded primitives
- Commit: `d72a88e`

**3. [Rule 3 тАФ Blocking] cargo build missing sidecar placeholders**
- Found during: Task 1 Step E verification
- Issue: `cargo build -p storycapture` failed in build.rs because `binaries/ffmpeg-aarch64-apple-darwin` and `binaries/playwright-sidecar-modules` did not exist (pre-existing dev-env state, not caused by this plan).
- Fix: Ran the already-documented `scripts/dev/install-sidecar-placeholders.sh` + created the `playwright-sidecar-modules` dir. Build now green. No scope expansion тАФ this was prerequisite dev-env setup, not code change.
- Commit: (none тАФ runtime-only placeholder files, gitignored)

**4. [Rule 2 тАФ Critical a11y] Label without htmlFor**
- Found during: Task 2 lint sweep
- Issue: biome `a11y/noLabelWithoutControl` flagged the stock shadcn Label signature (`React.ComponentProps<"label">`) because htmlFor is optional and the component has no child input.
- Fix: Typed `htmlFor` as required (`LabelProps extends LabelHTMLAttributes<HTMLLabelElement> { htmlFor: string }`). Call sites are forced to pair the label with a control. Root-cause fix per CLAUDE.md.
- Commit: `d72a88e`
